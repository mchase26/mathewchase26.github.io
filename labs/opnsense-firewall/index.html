NETWORK SECURITY LAB

OPNsense Firewall
Virtual Network Security Environment

Designed and configured a virtual firewall environment using OPNSense to practice network security, routing, NAT, DHCP, and firewall administration.

Lab Network Topology

Project Overview

The objective of this lab was to deploy OPNsense as a virtual firewall between an internal LAN network and the internet-facing WAN interface.


This project simulated a small business network where security policies, traffic filtering, and network services must be properly configured and maintained.

Lab Environment

Oracle Virtual Box
OPNSense Firewall
Linux Ubuntu Server
Windows 11 Client

Lab Screenshots and Description

OPNSense Firewall Terminal Startup

Installed OPNSense Firewall OS into Virtualbox. Terminal started up for firwall configuration.

WAN Configuration

Configured WAN and LAN in terminal. Statically assigned LAN IP address as the gateway to proper VLAN (192.168.10.1). WAN IP dynamically assigned to NAT adapter on VM (10.0.x.x).

Connectivity Confirmation

Confirmed successful LAN-to-WAN communication by pinging 8.8.8.8 and google.com, validating Internet connectivity and functional DNS resolution.

Windows 11 IP Assignment & Network Connectivity

Manually configured Windows 11 internet adapter to proper VLAN (192.168.10.10). used ipconfig command to verify IP address and Gateway. Verified connectivity by pinging 8.8.8.8 and Google.com.

Verify Connectivity on Windows Device

Used Ping to verify connectivity on the internet using 8.8.8.8 and Google.com.

Ubuntu Server Network Configuration

Configured yaml file in nano to assign static IP address and Default Gateway address. Ubuntu Server was put on VLAN 192.168.10.20/24.

Verify Ubuntu Server Connectivity

Used Ping to verify connectivity on the internet using 8.8.8.8 and Google.com.

Firewall GUI Configuration

Logged into the firewall's GUI on the Windows client. Blocked insecure ports on LAN side, such as Server Message Block (SMB port 445), Telnet (port 23), and Remote Desktop Protocol (port 3389).

Verify ports are being blcoked

Went into powershell and ran "Test-NetConnection" on the Ubuntu Server Host for ports 445 and 80. The cmdlet verifies that port 445 did NOT succeed a TCP test, but port 80 did, as the firewall rule has port 80 as "Allowed" in the rule set.

Apache Web Server Online

Installed and updated Apache Web Server on Ubuntu. Status shows that it is online and active via IP address 192.168.10.20.

Apache Web Server Online

Verified that Apache is online by accessing the GUI on the Windows host device via 192.168.10.20. All services (Apache, OPNSense, and Windows) are actively online and running in this state.

ICMP Packket Analysis

Verified that packets are moving throughout the network. Started with ICMP, we can verify that we see replies and requests from all devices on the network. Starting with OPNSense (192.168.10.1) to the Windows VLAN (192.168.10.10). We also verified the Apache Server (192.168.10.20) is receiving replies from the Windows machine (192.168.10.10).

Verify TCP Communication

In Wireshark, the packets captured a successfull HTTP communication between the Windows 11 Client and the Apache Web Server through the OPNSense Firewall. Traffice confirmed TCP Connectivity, HTTP Get Requests, and HTTP 200 OK responses while demonstrating packet inspection and network troubleshooting skills.

tcpdump analysis

I also used tcpdump to capture packets on the Apache Server side. The packet capture confirms successful TCP communication, HTTP GET requests, HTTP 200 OK responses from the Apache web server, and proper routing through the OPNsense firewall. This analysis demonstrates practical experience with packet inspection, protocol analysis, and troubleshooting enterprise network communications.

Incident Response Report

Severity: Informational

Status: Resolved

Root Cause: Initial WAN configuration prevented outbound connectivity and DNS resolution.

Impact: Internal hosts could not access external resources until the issue was corrected.

Resolution: Configured the WAN interface, verified gateway assignment, confirmed outbound NAT, and validated connectivity using ICMP and DNS tests.
